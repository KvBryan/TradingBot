================================================================================
DOCUMENTACIÓN GENERAL DEL PROYECTO: SISTEMA DE ALGO-TRADING HMM V6
================================================================================

Este documento detalla exhaustivamente la arquitectura, el diseño matemático, el stack tecnológico, la estructura y el flujo de despliegue del sistema cuantitativo automatizado de trading. Está estructurado de forma técnica para facilitar el análisis por modelos de Inteligencia Artificial u otros motores de desarrollo.

--------------------------------------------------------------------------------
1. OBJETIVO DEL SISTEMA
--------------------------------------------------------------------------------
El sistema está diseñado para la operación automatizada en el mercado Forex y commodities 24/7. Su propósito es capturar movimientos expansivos de tendencia y filtrar por completo los periodos de consolidación lateral ("chop" o rango plano) que causan la mayoría de pérdidas en estrategias tendenciales. 

La ejecución se realiza conectando las señales algorítmicas de TradingView en tiempo real con la API del broker Capital.com a través de un backend intermedio que valida la gestión de riesgos, almacena el historial en base de datos y provee visualización a través de un panel de control en tiempo real.

--------------------------------------------------------------------------------
2. ARQUITECTURA GLOBAL DEL SISTEMA (FLUJO DE DATOS)
--------------------------------------------------------------------------------
El sistema opera bajo un flujo lineal cerrado con retroalimentación:

[ TradingView (Pine Script v6) ]
               │
               ▼ (Alerta Webhook POST JSON con trade_id único)
[ Caddy Proxy Inverso (HTTPS / SSL) ]
               │
               ▼
[ Webhook Listener (ASP.NET Core Web API) ] <───> [ PostgreSQL Db ]
               │                                      │
               ├──────────────────────────────────────┤
               ▼ (REST API Orders)                    ▼ (SignalR WebSockets)
[ Capital.com Broker API ]               [ React Dashboard (Vite) ]
               │
               ▼ (Ejecución real)
        [ Mercado Forex ]

3. DETALLE DEL STACK TECNOLÓGICO
--------------------------------------------------------------------------------

A. ESTRATEGIA Y GENERACIÓN DE SEÑALES (FRONT-END TRADINGVIEW)
   - Compilador: Pine Script v6.
   - Parámetro de Regla de Cierre: `close_entries_rule = "ANY"` inyectado para asegurar la compatibilidad con el motor v6 y evitar la desincronización FIFO de posiciones acumuladas.
   - Motor Matemático: Hidden Markov Model (HMM) autocalibrable de 2 estados.
   - Lógica de Entrada: Canal Darvas Box adaptado sin retraso de pivote y ejecución por **Órdenes Límite** (Buy Limit / Sell Limit).
   - Lógica de Riesgo: Lotaje dinámico por volatilidad ATR y especificaciones del contrato (`syminfo.contractsize`, `syminfo.pipvalue`).

B. PROCESAMIENTO Y EJECUCIÓN (BACKEND - MICROSERVICIO C#)
   - Framework: .NET 8/9 (ASP.NET Core Minimal APIs).
   - Base de Datos: PostgreSQL.
   - ORM: Entity Framework Core (EF Core) con soporte para migraciones.
   - Broker API Client: HttpClient seguro que implementa cifrado y autenticación por API Keys de Capital.com (Demo/Live).
   - Hilos en Segundo Plano: HostedService (`CapitalComSyncWorker`) para sincronizar cuentas y posiciones activas del broker cada N segundos.
   - Comunicación en tiempo real: SignalR Hub (`TradeHub`) para notificar ejecuciones y telemetría al frontend.
   - Seguridad: JWT Bearer tokens para proteger los endpoints y las conexiones WebSocket del dashboard.

C. INTERFAZ GRÁFICA (FRONTEND - PANEL DE CONTROL)
   - Framework: React (Vite) + TypeScript + Tailwind CSS.
   - Conectividad: Cliente SignalR para suscripción a WebSockets y Axios para peticiones REST al Backend.
   - Funcionalidades: Telemetría de cuenta (Margen, Equidad, Balance), listado de posiciones activas, registro de logs de auditoría en vivo y estado diario del drawdown.

D. DESPLIEGUE E INFRAESTRUCTURA
   - Servidor: VPS Linux (Ubuntu).
   - Contenedores: Docker & Docker Compose para empaquetar el microservicio C# y la base de datos PostgreSQL.
   - Servidor Web & SSL: Caddy Server (configurado como proxy inverso y gestor automático de certificados Let's Encrypt).

5. DETALLE DE LA ESTRATEGIA CUANTITATIVA Y SOLUCIÓN DE VULNERABILIDADES
--------------------------------------------------------------------------------

A. FORMULACIÓN MATEMÁTICA DEL HIDDEN MARKOV MODEL (HMM)
   Para evitar el ruido en canales laterales, el script implementa un modelo analítico matricial recursivo que asume 2 regímenes de mercado:
     - Régimen 0 (Baja volatilidad / Rango plano).
     - Régimen 1 (Alta volatilidad / Tendencia expansiva).

   Fórmula de Observación (Retorno Logarítmico Estandarizado):
     R_t = (ln(C_t / C_{t-1}) - μ) / σ
     Donde el promedio (μ) y la desviación estándar (σ) se calculan dinámicamente sobre el periodo parametrizado (hmm_period).

   Autocalibración de Estados y Reducción de Sobreajuste (Overfitting):
     Para cumplir con las normas cuantitativas de robustez paramétrica (máximo 2-4 variables libres), el 70% de las variables se han quemado en el código (hardcoded) como constantes matemáticas. Las desviaciones estándar del HMM se adaptan dinámicamente respecto a la desviación a largo plazo (200 velas), eliminando variables libres de optimización:
     - σ0 = LongTermSigma * 0.5 (Estado 0 - Compresión)
     - σ1 = LongTermSigma * 1.5 (Estado 1 - Expansión)

   Probabilidades de Emisión Gaussiana para cada Estado S_i:
     P(Obs | S_i) = [1 / (σ_i * sqrt(2π))] * exp( - (R_t - μ_i)^2 / (2 * σ_i^2) )

   Matriz de Transición de Probabilidades (Fija):
     A = [ p00  p01 ] = [ 0.85  0.15 ]
         [ p10  p11 ]   [ 0.20  0.80 ]

   Algoritmo Forward (Actualización de Probabilidades):
     f0_new = emit0 * (f0 * p00 + f1 * p10)
     f1_new = emit1 * (f0 * p01 + f1 * p11)
     sum_f = f0_new + f1_new
     - f0 = f0_new / sum_f
     - f1 = f1_new / sum_f

B. MÓDULO DE ENTRADA: DARVAS BOX RETEST MEDIANTE ÓRDENES LÍMITE (0 LAG)
   - El canal de consolidación se define mediante el máximo y mínimo móvil de los últimos periodos (`darvas_period = 10`), con desfase de 1 vela `[1]` para evitar lookahead bias en el gráfico.
   - Ruptura: Ocurre cuando la vela actual cruza los niveles superior/inferior.
   - Orden Límite en TradingView: Al ocurrir una ruptura en TradingView, el script no espera confirmación de cierre de vela. Coloca una orden pendiente **BUY LIMIT** en el techo roto (o **SELL LIMIT** en el suelo roto):
     `strategy.entry("Long_Trade", strategy.long, limit=retest_level, qty=qty, ...)`
   - **Análisis de Latencia y Jerarquía de Estados:** TradingView actúa como el **Maestro de Estados de Salida**. Si colocáramos la orden límite directamente en el libro de órdenes del broker real antes de que se llene en TradingView, TradingView nunca sabría cuándo fue ejecutada (ya que Pine Script no tiene APIs de entrada para recibir notificaciones del broker). Al no registrar el llenado, TradingView no activaría su motor interno de exits (Break-Even a 1:1, Trailing ATR dinámico, protección diaria). Por ende, para que TV gestione las salidas, la orden límite se simula en TradingView; al ejecutarse la orden virtual, TradingView envía inmediatamente el webhook de compra/venta a C#, el cual lanza una orden a mercado en Capital.com. La latencia de red (milisegundos) se minimiza usando servidores en la nube y optimización de sockets.
   - Cancelación de Límites Obsoletos: Si el precio no regresa a retestear el nivel en menos de 15 velas (`darvas_max_retest_bars`), el script cancela la orden pendiente para evitar que se ejecute una señal vieja.

C. VALOR DINÁMICO DEL PIP Y CONVERSOR DE CONTRATOS (UNIVERSAL)
   Para poder operar Forex en JPY o cruces complejos, oro o materias primas sin que el riesgo de tu capital del 1% varíe:
     qty_units = CapitalArriesgado / (DistanciaStopPips * syminfo.pipvalue)
     lotes = qty_units / syminfo.contractsize

D. SOLUCIÓN A LA SERIALIZACIÓN JSON Y NULOS EN C#
   - Para evitar excepciones de mapeo `JsonException` o `FormatException` en el backend al enviar nulos (ej: en cierres o trailing stop sin Take Profit), el DTO de C# (`WebhookRequest`) define todos los campos numéricos como anulables (`decimal? StopLoss`, `decimal? TakeProfit`). 
   - El backend maneja de forma segura las variables mediante `.GetValueOrDefault()`.
   - En Pine Script, la función `f_build_json` concatena el literal `null` de JSON (sin comillas) cuando una variable tiene el valor `na`.

E. EVITAR ÓRDENES DUPLICADAS (IDEMPOTENCIA POR `trade_id`)
   - En Pine Script, el webhook incluye un identificador de transacción único `"trade_id"` generado dinámicamente combinando el Ticker, el número mágico del módulo y la marca de tiempo de la vela:
     `trade_id = Ticker_MagicNumber_Time`
   - En C# (`ReceiveWebhookEndpoint.cs`), se almacena cada `trade_id` procesado en una caché en memoria thread-safe (`ConcurrentDictionary<string, bool>`). Antes de pasar la orden a ejecución, el backend valida si el ID ya existe; si es duplicado (causado por reintentos de red), se ignora de inmediato el payload devolviendo `200 OK`.

F. PROTECCIÓN DE APERTURA DUPLICADA EN BROKER (PREVENCIÓN DE SOBREEXPOSICIÓN)
   - Adicionalmente a la validación de idempotencia, el backend en C# realiza una consulta en la base de datos PostgreSQL antes de abrir una operación.
   - Si se recibe una alerta de compra/venta (`BUY` / `SELL`) pero la base de datos detecta que ya existe una posición con estado `"OPEN"` para ese mismo símbolo y estrategia, la orden es abortada preventivamente. Esto evita acumular posiciones paralelas por desincronización.

6. DETALLE DEL WEBHOOK LISTENER Y PROTOCOLOS DE PRODUCCIÓN
--------------------------------------------------------------------------------

A. PROTECCIÓN CONTRA SPREAD DINÁMICO EN BROKER
   - Se removió el control de spread máximo del código de TradingView porque la plataforma opera con datos estáticos e idealizados.
   - **Protocolo de Producción (Backend):** Cuando el microservicio recibe la señal de compra/venta del Webhook de TradingView, antes de enviar la orden de ejecución a Capital.com realiza una llamada de consulta rápida del libro de órdenes (*order book*) de Capital.com para verificar el spread real de mercado. Si el spread real en ese microsegundo es superior al spread máximo permitido, el backend rechaza la orden y registra el log correspondiente de bloqueo preventivo.

B. CONCILIACIÓN AUTOMÁTICA DE ESTADO
   - Si la posición es cerrada manualmente por el usuario desde el panel del broker o salta el Stop Loss físico, el `CapitalComSyncWorker` en segundo plano actualiza la base de datos PostgreSQL a `"CLOSED"`.
   - Si TradingView envía tardíamente un webhook de cierre (`CLOSE`) sobre un trade ya cerrado en el broker, el backend intercepta la llamada, actualiza la base de datos local y devuelve un código de éxito `200 OK` al webhook de TradingView sin lanzar errores ni intentar cerrar órdenes inexistentes en Capital.com.

9. CONTRATO DE INTEGRACIÓN PINE <-> BACKEND
--------------------------------------------------------------------------------
Este contrato define formalmente la interfaz de comunicación entre las alertas generadas por TradingView (Pine Script v6) y el backend receptor (C# / .NET).

A. Protocolo y Endpoint:
   - Método: HTTP POST.
   - Endpoint: `https://[dominio]/api/v1/webhook` (ej. `https://bot.kvstudio.me/api/v1/webhook`).
   - Content-Type: `application/json` o texto plano legible como JSON.

B. Ejemplo de Payload JSON (Apertura):
   ```json
   {
     "action": "BUY",
     "symbol": "EURUSD",
     "lot_size": 1.45,
     "entry_price": 1.08250,
     "stop_loss": 1.08050,
     "take_profit": 1.08650,
     "strategy": "Darvas Box",
     "magic_number": 1001001,
     "trade_id": "EURUSD_1001001_1717464000000",
     "timestamp": 1717464000000
   }
   ```

C. Ejemplo de Payload JSON (Cierre):
   ```json
   {
     "action": "CLOSE",
     "symbol": "EURUSD",
     "lot_size": 1.45,
     "entry_price": 1.08250,
     "stop_loss": 1.08050,
     "take_profit": null,
     "strategy": "Darvas Box",
     "magic_number": 1001001,
     "trade_id": "EURUSD_1001001_1717464000000",
     "timestamp": 1717464000000
   }
   ```

D. Campos Obligatorios y Validaciones:
   - "action" (string): Obligatorio. Valores admitidos: "BUY", "SELL", "CLOSE", "CANCEL".
   - "symbol" (string): Obligatorio. Identifica el instrumento (ej. "EURUSD", "GBPUSD", "USDJPY").
   - "entry_price", "stop_loss", "take_profit" (decimal/null):
     - Para aperturas (BUY / SELL): Son estrictamente obligatorios y deben ser mayores a cero.
     - Para cierres (CLOSE / CANCEL): Se pueden omitir o enviar como null.
   - "strategy" (string): Obligatorio. Identifica la estrategia de procedencia (ej. "Darvas Box").
   - "trade_id" (string): Obligatorio. Hash único en formato: `Ticker_MagicNumber_TimestampUnix`.

E. Mecanismo de Autenticación y Seguridad del Webhook:
   - El webhook no utiliza JWT directamente en la cabecera (el JWT se reserva para el dashboard web y SignalR).
   - Seguridad Perimetral: Se configura a nivel de proxy inverso (Caddy / Nginx) limitando las peticiones entrantes al endpoint `/api/v1/webhook` únicamente a los rangos de direcciones IP públicas oficiales de TradingView.
   - Protección contra Replays y Duplicados: El backend implementa una validación estricta de idempotencia en memoria mediante `ProcessedTradeIds`. Si el `trade_id` ya fue procesado, se rechaza de inmediato (retornando éxito al emisor para evitar reintentos fallidos de red).

F. Códigos de Respuesta Esperados:
   - `200 OK` con `{ "success": true, "message": "..." }`: Procesamiento y ejecución exitosa de la orden.
   - `200 OK` con `{ "success": false, "message": "..." }`: El webhook fue estructurado de forma válida pero la orden fue rechazada preventivamente (ej. por trade duplicado detectado en PostgreSQL o por idempotencia de `trade_id`).
   - `400 Bad Request` con `{ "success": false, "message": "..." }`: JSON mal formado o ausencia de parámetros obligatorios en la validación inicial (por ejemplo, omitir stop loss en una apertura).

10. DIAGRAMA DE ESTADOS DE LA SEÑAL
--------------------------------------------------------------------------------
El ciclo de vida de una señal operativa se describe mediante la siguiente jerarquía de estados:

```mermaid
stateDiagram-v2
    [*] --> Idle : Monitorizando Mercado (Barra a Barra)
    
    Idle --> Breakout_Detected : Ruptura de Darvas Box (Top/Bottom)
    note right of Breakout_Detected
        Se activa waiting_retest = true
        Se define retest_level = nivel roto
        retest_bars_counter = 0
    end note

    Breakout_Detected --> Idle : Expiración (retest_bars_counter > 15 velas) o ruptura opuesta
    
    Breakout_Detected --> Limit_Order_Placed : allowed_to_enter == true
    note right of Limit_Order_Placed
        Se coloca orden límite virtual en TradingView.
    end note
    
    Limit_Order_Placed --> Idle : Cancelación de orden (retest_bars_counter > 15 velas)
    
    Limit_Order_Placed --> Position_Entered : Precio retestea y toca el nivel
    note right of Position_Entered
        Se ejecuta orden límite en TradingView.
        Se envía Webhook POST al backend.
        Backend abre orden real a mercado en Capital.com.
        Estado del trade en PostgreSQL pasa a "OPEN".
    end note
    
    Position_Entered --> Break_Even_Active : Precio alcanza RRR 1:1 (be_trigger_price)
    note right of Break_Even_Active
        El SL se ajusta al precio de entrada más buffer de spread.
    end note
    
    Position_Entered --> Trailing_Active : Precio alcanza 1.5 RRR
    Break_Even_Active --> Trailing_Active : Precio alcanza 1.5 RRR
    note right of Trailing_Active
        Take Profit estático se anula (tp = null).
        Trailing Stop ATR dinámico se activa (1.5 * ATR14 bar-by-bar).
    end note
    
    Position_Entered --> Position_Closed : Precio toca Stop Loss o Take Profit estático
    Break_Even_Active --> Position_Closed : Precio toca Stop Loss en BE
    Trailing_Active --> Position_Closed : Precio toca Trailing Stop
    
    Position_Closed --> Idle : Webhook de salida recibido en Backend. DB se actualiza a "CLOSED".
```

11. CONVENCIÓN TEMPORAL Y ZONAS HORARIAS
--------------------------------------------------------------------------------
Para garantizar la consistencia en el cálculo de drawdown diario, el control horario y las marcas de tiempo entre el gráfico de TradingView, el microservicio y el broker, se establecen las siguientes reglas:

- **Reset del Drawdown Diario**: Se ejecuta mediante `ta.change(time("D")) != 0` utilizando la zona horaria del instrumento (Forex operando 24 horas usa UTC). El drawdown diario y el contador de trades se reinician a cero a las 00:00 UTC.
- **Sesión Operativa Nueva York (NY)**: La ventana operativa segura de volatilidad se define mediante el parámetro `ny_session = "0600-1200:23456"`, evaluado en base al huso horario de Nueva York (EST/EDT) para concentrar las entradas en las horas de mayor liquidez.
- **Filtro de Noticias Macro**: Los límites de bloqueo de noticias (`news1_start`, etc.) se declaran utilizando el timezone local de New York especificando el desfase GMT en Pine Script (ej. `-0400` para horario de verano de Nueva York - EDT).
- **Marcas de Tiempo (Timestamps)**: El webhook envía la variable nativa `time` de Pine (milisegundos Unix en formato UTC). El backend almacena todas las marcas de tiempo de creación e histórico en PostgreSQL utilizando UTC (`DateTime.UtcNow`).

12. DETALLE DE PARÁMETROS MATEMÁTICOS DEL HMM (JUSTIFICACIÓN DE CONSTANTES)
--------------------------------------------------------------------------------
Para cumplir con los principios cuantitativos de diseño robusto y blindar el bot contra el sobreajuste (curve-fitting), el modelado del Hidden Markov Model (HMM) fija por completo en el código el 70% de las variables. La justificación de estos valores constantes es:

- `hmm_long_period = 200`: Calibra la desviación estándar estructural de largo plazo (`long_term_sigma`). Una ventana de 200 barras representa la volatilidad promedio histórica e impide que picos de volatilidad aislados e inestables sesguen el cálculo dinámico de estados del HMM.
- `mu0 = 0.0` y `mu1 = 0.0`: En retornos logarítmicos financieros de corto a mediano plazo, el retorno promedio esperado tiende matemáticamente a cero. Estimar y calibrar medias dinámicas sobre muestras pequeñas añade ruido e inestabilidad al sistema.
- `sigma0_mult = 0.5`: Multiplicador para el Estado 0 (Compresión de volatilidad). Fija que el estado de compresión/rango tiene una desviación equivalente a la mitad de la volatilidad histórica a largo plazo, aislando eficazmente periodos laterales.
- `sigma1_mult = 1.5`: Multiplicador para el Estado 1 (Expansión de tendencia). Define que el estado tendencial requiere de un aumento del 150% en la desviación de los retornos logarítmicos con respecto al histórico base, protegiendo al bot de rupturas falsas.
- **Matriz de Transición Fija ($A$)**:
  - $p_{00} = 0.85$, $p_{01} = 0.15$: Representa que los mercados permanecen en rango o consolidación el 85% del tiempo.
  - $p_{11} = 0.80$, $p_{10} = 0.20$: Permite la inercia tendencial una vez iniciado el impulso expansivo, pero fuerza una transición rápida a compresión si la volatilidad decrece.
  - *Justificación general*: Al fijar estos coeficientes de probabilidad, la estimación del régimen HMM se vuelve dependiente únicamente de un parámetro libre (`hmm_period`). Esto cumple con las restricciones paramétricas ideales (máximo 2-4 variables libres en total) necesarias para superar backtests cuantitativos exigentes.

13. SEMÁNTICA DEL MAGIC NUMBER DINÁMICO
--------------------------------------------------------------------------------
El número mágico se compone en Pine Script de forma dinámica mediante la fórmula:
`dynamic_magic = active_magic * 1000 + daily_trade_count`

Donde:
- `active_magic` (int): Identifica el sentido y origen de la orden.
  - `1001`: Aperturas de Compra (Long).
  - `1002`: Aperturas de Venta (Short).
- `daily_trade_count` (int): Secuencia interna de operaciones abiertas en el día calendario actual (comienza en 0 tras el reset diario y se incrementa en 1 por cada orden ejecutada).

**Uso aguas abajo (Downstream en Backend)**:
- **Idempotencia avanzada**: El backend guarda el número mágico en PostgreSQL. En caso de recibir un webhook de cierre (`CLOSE`) para un par que tiene múltiples ejecuciones simultáneas debido a otras estrategias o cuentas paralelas, la combinación del magic_number junto con la estrategia permite reconciliar e identificar cuál posición exacta debe cerrarse en Capital.com.
- **Auditoría e Historial**: El número mágico permite la reconstrucción cronológica y análisis estadístico diario en la base de datos sin depender únicamente de marcas de tiempo del servidor.

14. ACLARACIONES E INCONSISTENCIAS DE DOCUMENTACIÓN RESUELTAS
--------------------------------------------------------------------------------
1. **Cálculo de Lotaje Dinámico vs ATR**:
   - *Corrección*: Se aclara que la posición de volumen (lotes) se calcula exclusivamente usando el capital arriesgado del 1%, la distancia al Stop Loss estructural en pips y el valor por contrato (`syminfo.pipvalue` / `syminfo.contractsize`). El ATR no influye en el cálculo del lotaje; el ATR se utiliza únicamente para el Stop de arrastre (Trailing Stop) dinámico.
2. **Filtro de Noticias Macro**:
   - *Aclaración*: El filtro es semi-manual. Depende de las variables `newsX_start` y `newsX_end` configuradas en Pine Script. Si no se actualizan las fechas para eventos futuros o si `news_expired` marca `true` (al pasar `news3_end`), el filtro debe desactivarse manualmente o actualizarse en TradingView para evitar el bloqueo preventivo indefinido.
3. **Interacción entre Filtros y Retest**:
   - *Aclaración*: El bot detecta las rupturas Darvas y activa el estado de retest en Pine Script *antes* de comprobar los filtros de entrada (`allowed_to_enter`). Si el precio retestea el nivel de retest en un momento en que los filtros temporales (ej: noticias o fuera de hora) ya no bloquean la entrada, el bot colocará la orden. Si los filtros siguen activos al retestear, la señal expira silenciosamente sin colocar orden límite virtual.
